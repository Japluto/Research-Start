### 🚀 ROS 1 Docker 环境启动全流程

#### 1. 宿主机准备（每次开机/重启后执行一次）

这一步是为了允许 Docker 容器把画面“投射”到你的物理屏幕上。

```bash
xhost +local:root

```

#### 2. 启动容器（后台模式）

进入你存放 `docker-compose.yml` 的目录，并启动服务。

```bash
cd ~/ros_docker
sudo docker compose up -d

```

*(注：如果之前未安装 V2 插件，可能需要用 `sudo docker-compose up -d`)*

#### 3. 进入容器（开始干活）

这一步相当于打开了一个“虚拟终端”，你在里面做的所有操作都是隔离环境的。

```bash
sudo docker exec -it ros_noetic_gpu bash

```

---

### 🛠️ 容器内验证与开发

进入容器后（提示符变更为 `root@...`），按照以下逻辑操作：

#### 1. 显卡与环境自检

由于我们已经把 `source /opt/ros/noetic/setup.bash` 写入了 `.bashrc`，登录即生效。

```bash
# 检查显卡是否被识别
nvidia-smi

# 检查环境变量（如果不报错，说明 source 成功）
roscd

```

#### 2. 运行 GUI 程序

推荐的工作流是：先在后台运行核心，再启动界面。

```bash
# 1. 后台启动 roscore
roscore &

# 2. 启动 Rviz 查看 3D 界面
rosrun rviz rviz

```

#### 3. 代码开发（关键！）

你的代码应该存放在**宿主机**的 `~/ros_docker/catkin_ws/src` 中。

* **写代码**：直接在 Ubuntu 宿主机用 VSCode 修改文件。
* **编译运行**：在 Docker 容器内执行：
```bash
cd /root/catkin_ws
catkin_make
source devel/setup.bash  # 编译后记得刷新工作空间环境

```



---

其实在实际开发中，我们反而**不建议**频繁地杀掉（Remove）容器，除非你想要一个全新的环境。

你可以把 Docker 容器的生命周期想象成一台电脑：

1. **`docker compose down` (杀掉)** = **“重装系统并关机”**。
* 容器会被彻底删除。
* 下次启动时，是基于镜像重新创建一个全新的容器。
* **后果**：你在容器里临时安装的软件（比如手动 `apt install` 的包）、修改的系统配置（非挂载卷内的），全都会**丢失**。


2. **`docker compose stop` (停止)** = **“电脑睡眠/休眠”**。
* 容器只是停止运行，进程结束，占用资源释放（显卡、CPU、内存归还给宿主机）。
* 但容器的**文件系统状态被保留**在硬盘上。
* **后果**：下次启动，所有东西都在，你可以接着上次的进度继续干。

### 推荐的日常开发流程（“睡眠”模式）

如果你明天还想接着用这个环境，**不要用 `down**`，请改用以下流程：

#### 1. 结束工作时：只停止，不删除

当你想下班或休息时：

```bash
sudo docker-compose stop

```

*此时，容器还在，只是状态变成了 `Exited`。你的显卡和内存已经被释放了。*

#### 2. 第二天开始工作：唤醒容器

```bash
sudo docker-compose start

```

#### 3. 再次进入容器

```bash
sudo docker exec -it ros_noetic_gpu bash

```

---

### 什么时候必须用“杀掉” (`down`)？

虽然日常用 `stop` 很方便，但在以下几种情况，你需要执行 `sudo docker compose down` 来彻底重置：

1. **修改了配置文件**：你修改了 `docker-compose.yml` 文件（比如新增了一个挂载目录，或者修改了环境变量），必须 `down` 然后再 `up` 才会生效。
2. **系统被玩坏了**：你在容器里乱删文件或者装错了驱动，导致环境报错，想“恢复出厂设置”。
3. **节省硬盘空间**：如果你几个月都不打算用这个项目了，删掉容器可以释放一些硬盘空间（容器层也会占用空间）。

### 总结建议

* **日常开发（短暂停）**：用 `stop` 和 `start`。这样你在容器里临时装个 `vim` 或者 `git` 它是能记住的。
* **环境重构（彻底清理）**：用 `down`。
* **代码保存**：无论你杀不杀容器，只要你的代码在 **`~/ros_docker/catkin_ws`**（宿主机挂载目录）里，它永远是安全的。

### 🛑 结束工作

当你做完实验，想要完全关闭环境释放资源时：

```bash
# 在宿主机 ~/ros_docker 目录下
sudo docker-compose down

```

*(这会停止并移除容器，但你的代码文件因为挂载在本地，**绝对安全**，不会丢失。)*
